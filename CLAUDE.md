# CLAUDE.md

This file documents the intended behavior of `visitor-ip-api-go` so future AI or human changes do not accidentally reintroduce complexity or change the API contract.

## Project Purpose

`visitor-ip-api-go` is a small Go HTTP API that returns:

- the best-known client IP for the current request

This service is intentionally simple. It is not a security product and it does not attempt to prove header authenticity.

## Core Design Decision

Use a simple best-effort client IP selection strategy.

Do not:

- validate Cloudflare edge IP ranges
- maintain trusted proxy CIDR lists
- treat proxy headers as a security boundary
- reject or ignore proxy headers because they could be spoofed

Reason:

- this project is meant to report the best-known client IP
- it must work behind Cloudflare, nginx, and other common reverse proxies
- operational simplicity is preferred over strict trust validation

## Client IP Resolution Order

The current logic is implemented in `internal/clientinfo/clientinfo.go`.

Resolution order:

1. `CF-Connecting-IP`
2. `CF-Connecting-IPv6` only for Cloudflare Pseudo IPv4 overwrite handling
3. first valid IP in `X-Forwarded-For`
4. `X-Real-IP`
5. `RemoteAddr`

More specifically:

1. Read `CF-Connecting-IP`
2. If it parses as an IP:
   - if it is inside `240.0.0.0/4` and `CF-Connecting-IPv6` contains a valid IPv6, return the IPv6
   - otherwise return `CF-Connecting-IP`
3. If Cloudflare headers do not yield an IP, parse `X-Forwarded-For`
   - use the first valid IP in the comma-separated list
4. If that fails, parse `X-Real-IP`
5. If that fails, parse `RemoteAddr`
6. If nothing parses, return an empty IP string

## Important Behavioral Rules

These rules are intentional and should not be changed casually.

### 1. Proxy headers are preferred over `RemoteAddr`

This is the main project policy.

If the service is behind nginx, Cloudflare, Caddy, Traefik, or another proxy, the API should prefer the forwarded client IP rather than returning the proxy IP from `RemoteAddr`.

### 2. No trusted-proxy enforcement

Do not reintroduce:

- `TRUST_CLOUDFLARE_HEADERS`
- `REQUIRE_CLOUDFLARE_EDGE`
- Cloudflare CIDR checks
- trusted proxy CIDR configuration

Those approaches were intentionally removed to keep the service straightforward and deployment-friendly.

### 3. Cloudflare pseudo IPv4 handling must stay

Cloudflare may overwrite `CF-Connecting-IP` with a pseudo IPv4 address in `240.0.0.0/4`.

When that happens and `CF-Connecting-IPv6` is available, the API must prefer the real IPv6.

This behavior is important and should be preserved.

## Current Configuration Surface

Only one runtime environment variable is intended for the app itself:

- `LISTEN_ADDR`

Do not add back older trust-related environment variables unless the project requirements explicitly change.

## Testing Expectations

If IP extraction logic changes, update or add tests in `internal/clientinfo/clientinfo_test.go`.

At minimum, tests should continue to cover:

- direct `RemoteAddr` fallback
- normal `CF-Connecting-IP`
- Cloudflare pseudo IPv4 overwrite with `CF-Connecting-IPv6`
- `X-Forwarded-For`
- `X-Real-IP`
- invalid forwarded headers falling back to `RemoteAddr`

Run:

```bash
go test ./...
```

## Future Change Rules

If modifying this project later, keep these constraints in mind:

- prefer simpler behavior over more “secure-looking” but deployment-hostile logic
- do not add trusted proxy validation unless requirements explicitly demand it
- do not change IP source precedence without updating both tests and docs
- do not remove Cloudflare pseudo IPv4 handling
- do not reintroduce response `$schema` or `checked_at` unless requirements explicitly change
- if adding new response fields, document them here and in `README.md`
- if renaming response fields, update OpenAPI, tests, README, and this file together

## Non-Goals

This project does not currently attempt to:

- verify that headers were added by Cloudflare
- verify that headers were added by nginx
- determine whether a user has both IPv4 and IPv6 connectivity
- perform active reachability checks
- act as an auth or security control
